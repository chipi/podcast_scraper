# Acceptance Test Configs

Configs in this directory are used by the **acceptance test runner** to run full pipeline runs
(RSS → transcript → summary → GI → KG → semantic index) with different stacks.

For runner flags, session layout, and analysis, see **`scripts/acceptance/README.md`**.

## Directory layout

| Path | Role |
| ---- | ---- |
| **`README.md`** | This file |
| **`FAST_CONFIG.yaml`** | Fast matrix: `defaults` + `runs` (`id`, `profile`, `feeds_shape` or `feeds` path). Used by **`FROM_FAST_STEMS=1`** / **`FAST_ONLY=1`**. |
| **`fragments/acceptance_defaults.yaml`** | Keys merged for **every** matrix row before `profile:` and feeds (harness, `max_episodes`, `multi_feed_strict`, …). |
| **`fragments/feeds_single.yaml`** | Placeholder `rss:` — rewritten under **`USE_FIXTURES=1`**. |
| **`fragments/feeds_multi.yaml`** | Five placeholder `feeds:` URLs — fixture slots **p01–p05**. |

Other **`*.yaml`** files under **`config/acceptance/`** are **not** part of the tracked matrix unless you choose to commit them.

## Fast matrix (CI)

**`FAST_CONFIG.yaml`** lists **enabled** rows. For each row the runner **runtime-merges**
`fragments/acceptance_defaults.yaml` + the feeds fragment + **`profile:`** (official bundle under
`config/profiles/`), injects an autogenerated **`run_id`**, and writes
**`sessions/session_*/materialized/{id}.yaml`**. **`Path.stem`** equals matrix **`id`** so
**`FAST_ONLY=1`** filtering works.

- **`enabled: false`** on a row skips it without deleting the entry.
- **No** second fast-list file: edit the tracked matrix (or use a branch).

**Full fast matrix + fixtures** (offline RSS, mock APIs; CI on main/release):

```bash
make test-acceptance-fixtures-fast
```

Equivalent flags:

```bash
make test-acceptance FROM_FAST_STEMS=1 USE_FIXTURES=1 \
  NO_AUTO_ANALYZE=1 NO_AUTO_BENCHMARK=1 TIMEOUT=1500
```

**`USE_FIXTURES=1`** rewrites placeholder hosts to the in-process E2E server; provider HTTP goes to
mock bases — no real billed calls on CI for that job.

### Multi-feed layout

Multi-feed configs use **`feeds:`** (alias **`rss_urls`**). Corpus layout is
`<output_dir>/feeds/rss_<host>_<hash>/` per feed. With **`USE_FIXTURES=1`**, the runner replaces
external placeholder URLs with distinct local fixture feeds.

For operator examples (profile + feeds spec), see **`docs/api/CONFIGURATION.md#multi-feed-compose`**
and **`config/examples/feeds.spec.example.yaml`**.

### Optional corpus topic clustering (RFC-075)

After a successful run with **`vector_search: true`**, you can build **`search/topic_clusters.json`**:

```bash
python -m podcast_scraper.cli topic-clusters --output-dir <corpus_parent>
```

See [RFC-075](../../docs/rfc/RFC-075-corpus-topic-clustering.md) and
[Semantic Search Guide](../../docs/guides/SEMANTIC_SEARCH_GUIDE.md).

## CI: GIL vs FAISS offset check after the fast matrix

On **push** to `main` / release branches, `.github/workflows/python-app.yml` runs
`make test-acceptance-fixtures-fast`, then **`make verify-gil-offsets-after-acceptance`**.
That walks the latest acceptance **`session_*/runs/run_*`** trees and runs
`verify-gil-chunk-offsets --strict` on every run that has **`search/metadata.json`**
(RFC-072 / issue #528).

```bash
make verify-gil-offsets-after-acceptance
```
