# Real-podcast (67-min) transcription + summary smoke (#959)

**Date:** 2026-06-11
**Issue:** #959
**Scope:** smoke validation of the v2-fixture-derived recommendations
against one real podcast episode. Not the full multi-episode sweep
the original #959 spec called for — see the "What this is NOT"
section at the bottom.

## TL;DR

- **Transcription quality and throughput numbers from v2 fixtures
  generalize to real audio within reasonable bounds.** MPS and DGX
  whisper-openai both produce usable transcripts at their v2-predicted
  realtime multiples; per-word disagreement vs the production
  pipeline reference is in the 13-18% range, consistent with normal
  whisper variance on long-form noisy audio (sponsor reads,
  music interludes, naturalistic disfluencies).
- **The #929 production routing recommendations hold on real audio**:
  whisper-openai on DGX is faster (5.2×) with a small per-word
  divergence cost (+0.05 WER) vs MPS on laptop (1.5×). For batch
  transcription DGX wins on throughput; for clean local laptop work
  MPS stays competitive.
- **Summary pipeline handles the 17,560-prompt-token / 14,850-word
  transcript cleanly** — Ollama qwen3.5:35b on the DGX produced a
  693-token summary in 12.4 s.

## Inputs

- **Episode**: `0007 - Sergey Levine - Building LLMs for Robotics`
  from the "Invest Like the Best" feed (`rss_feeds.megaphone.fm_3581c092`).
- **Audio**: 67-minute MP3 (62 MB, 4071 s).
- **Reference transcript**: 14,850 words, 81 KB. Produced by the
  production pipeline (`rediar-batch-iltb` run, 2026-06-10).
- **Run dir**: `data/eval/runs/whisper_dgx_vs_cloud_v1_real90/`.

Note: the "reference" is itself a backend output (most likely Cloud
Whisper via the operator's prod-routing config), not a
human-corrected gold transcript. WER numbers are therefore
**vs-production-transcript**, not vs-ground-truth. The size of the
disagreement is meaningful (Δ between backends), but the absolute
WER is biased toward whatever the reference's stack happened to
produce.

## Numbers — transcription

| Backend | Model | WER (vs pipeline ref) | Wall (s) | Realtime × | Hyp words | Ref words |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| MPS (laptop) | large-v3 | **0.132** | 2,709 (45 min) | 1.5× | 15,235 | 14,850 |
| DGX whisper-openai (`:8002`) | large-v3 | **0.182** | 789 (13 min) | 5.2× | 14,245 | 14,850 |
| Cloud Whisper API | whisper-1 | (not run — 25 MB upload limit; would need ffmpeg chunking) | — | — | — | — |
| DGX faster-whisper (`:8000`) | large-v3 (int8) | (not run — blocked on #957 / #968 — int8 produces text but bimodal quality on the v2 set) | — | — | — | — |

### What the numbers tell us

1. **MPS is the closest to the pipeline reference** (WER 0.132). If
   the production reference was produced by Cloud Whisper (most
   likely given the v2 hybrid-routing config), this says MPS's
   `openai-whisper` library outputs the most Cloud-like text.
   Reasonable: same library family.
2. **DGX whisper-openai diverges more** (WER 0.182), but produces
   ~600 fewer words than the reference (96% of reference length).
   This shape suggests DGX is more aggressive at filtering
   non-speech / sponsor-read style content, or more conservative
   on disfluencies. Neither is "wrong" — different prompting + VAD
   defaults across containers.
3. **Throughput scales linearly with audio length** — both backends'
   realtime multiples on this 67-min episode match their v2-fixture
   numbers (MPS 1.5× vs v2 1.6×; DGX 5.2× vs v2 4.6×). No
   autoregressive slowdown at length. The temperature-schedule fix
   from #929 holds on long-form audio.
4. **Word counts agree within ~7% across all three** (ref 14,850 /
   MPS 15,235 / DGX 14,245). Both backends are producing
   substantively the same transcript, just with different
   word-boundary + sponsor-read decisions.

### What WER ~13-18% means on real audio

For context vs. v2 fixtures:

| Audio | MPS WER (vs ref) | DGX WER (vs ref) | Reference quality |
| --- | ---: | ---: | --- |
| v2 fixture (synth, 5 min) | 0.096 | 0.102 | human-corrected gold |
| Real 67-min episode | 0.132 | 0.182 | another whisper output |

The jump from ~0.10 to ~0.15 is consistent with:

- Real audio noise (mic plosives, ambience, music interludes).
- Sponsor-read passages where backends differ on whether to
  transcribe brand names verbatim.
- Naturalistic disfluencies ("um", "uh", overlap) handled
  differently across whisper variants.
- The reference being a backend output, not a gold transcript —
  some of the divergence is the reference's own error budget.

None of this is unexpected or production-blocking. The v2-derived
WER tier system (~10% on synth → expect ~15-20% on real, with
ranking preserved) holds.

## Numbers — summary

The reference transcript (14,850 words / 17,560 prompt tokens) was
summarized via Ollama `qwen3.5:35b` on the DGX, using the same
prompts the production hybrid-routing config uses
(`prompts/ollama/qwen3.5_35b/summarization/{system,long}_v1.j2`).

| Backend | Latency | Prompt tokens | Output tokens | Output chars |
| --- | ---: | ---: | ---: | ---: |
| Ollama qwen3.5:35b | **12.4 s** | 17,560 | 693 | 4,302 |

### What the summary numbers tell us

- **Real-length transcripts fit comfortably in qwen3.5:35b's
  context window** — 17.5 K prompt tokens is well under the
  configured limit. No chunking required.
- **12 s of wall time per real episode** scales to a production
  batch (~100 eps per run) at ~20 min total summary time. Fine.
- **The first 400 chars of the summary read cleanly** — the model
  identified Sergey Levine's "general-purpose robotic foundation
  models" thesis as the main takeaway. No reasoning-preamble leak
  (qwen3.5 with `think=false` behaves; the #961 prompt-engineering
  fix is R1-Distill-specific).
- Notably, the first call attempt without `think=false` returned
  EMPTY content — qwen3.5 is a "thinking" model, so without
  `think=false` it spends its `num_predict` budget on hidden
  reasoning tokens. **`think=false` is load-bearing for batch
  summary calls.** Our production path is using the OpenAI-compatible
  endpoint (which has its own template handling); the bare Ollama
  `/api/chat` path needs the explicit flag.

## What this is NOT (vs the original #959 spec)

The #959 issue called for a multi-episode (2-3) × multi-backend
(MPS + DGX whisper-openai + DGX faster-whisper + Cloud Whisper)
transcription sweep, followed by a 3-backend summary comparison
(Ollama qwen3.5:35b + vLLM Qwen3.6 + gemini-2.5-flash-lite) and a
finale judge round.

This smoke is **one episode × 2 transcription backends × 1 summary
backend** — about 1/12th of the full sweep. Scoping back reasons:

1. **Time** — the full sweep is ~6 hours of wall-clock per the
   #959 spec; this session had ~90 min budget for phase 3.
2. **Cloud Whisper API has a 25 MB upload limit** — 62 MB MP3
   would need ffmpeg chunking + a multi-call assembly step.
   Doable, but separate scope.
3. **DGX faster-whisper is blocked on #957 / #968** — int8 produces
   text but quality is bimodal on v2 (0.05 / 0.60 WER). Not safe
   for a real-audio measurement until the #968 Thread B
   temperature-fallback fix lands.
4. **The harness only saves WER metrics, not the hypothesis
   transcripts** — running 3-backend summary requires re-running
   transcription with output capture. Out of scope for this smoke.

What this smoke DOES validate is the cheapest, highest-value piece
of the #959 question — "do the v2 numbers generalize to real
audio?" — and the answer is "yes, within reasonable bounds."

## Follow-ups (recommended scope for the full #959 run)

1. **Patch the harness to save the hypothesis transcripts**
   (just write `<ep>.hyp.txt` alongside `metrics.json` in the
   output dir). One-line change in `whisper_dgx_vs_cloud_v1.py`.
   Cheap unblocker for the summary side of the sweep.
2. **ffmpeg-chunk + reassemble for Cloud Whisper API**. Several
   public recipes; would let us close the 4-way story on real audio.
3. **Re-run on 2-3 more episodes** for stat-confidence. Bigger
   variance per-episode on real audio than on v2 fixtures means
   one episode is enough for a "do the numbers hold" check but
   not for tight Δ measurement.
4. **Add a 3-backend summary leg** once the hyp transcripts are
   saved: Ollama qwen3.5:35b + vLLM Qwen3.6 + gemini-2.5-flash-lite,
   finale judges, see if the v2 summary rankings hold on real
   transcripts.
5. **DGX faster-whisper re-test** once #968 Thread B / C land —
   the int8 + temperature-fallback + VAD-upgrade combination is
   the question that closes the 4-way story.
