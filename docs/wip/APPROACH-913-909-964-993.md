## #993 — ElevenLabs as a fixture-audio TTS backend → **smallest real task**

Everything lives in one standalone tool: `tests/fixtures/scripts/transcripts_to_mp3.py` (the #934 fixture-audio generator). It is **not** part of `src/` and not in the runtime pipeline — purely for generating test fixtures.

**Current design:** conditional dispatch (`--backend say|gemini`), no abstraction. `say` = deterministic macOS offline (default); `gemini` = cloud multi-speaker. Each backend = a voice-map dict + a `get_*_voice_for_speaker()` + a `render_segments_via_*()` + a dispatch branch + an env-var key.

**Plan (drop-in, mirror the gemini backend):**
1. `SPEAKER_ELEVENLABS_VOICE_MAP` + fallback voice list (ElevenLabs voice IDs).
2. `get_elevenlabs_voice_for_speaker(name)` — same name→voice→hash-fallback shape as gemini.
3. `render_segments_via_elevenlabs(segments, out_mp3, bitrate, model, api_key)` — per-segment TTS (ElevenLabs has no native multi-speaker), decode → mono 16-bit WAV → ffmpeg concat → MP3, matching the existing format contract (mono, 16-bit, 64k default, `tests/fixtures/audio/<version>/`).
4. CLI: add `"elevenlabs"` to `--backend` choices + `--elevenlabs-model`.
5. Env: `ELEVENLABS_API_KEY` (validate-or-exit-2, same as `GEMINI_API_KEY`).
6. Dispatch branch + docstring.

**Decisions to make:**
- **Do NOT refactor to a Protocol/factory now** — it's a 3rd backend in a throwaway tool; the conditional dispatch is fine. (Refactor only if a 4th lands.) Keeps churn minimal.
- `elevenlabs` needs the `elevenlabs` SDK (or raw HTTP) — add to a fixture-only dependency note, **not** to a runtime extra. Confirm whether to use the SDK or `requests` (lean toward `requests` to avoid a new dep).
- It's a paid, non-deterministic cloud backend → it stays an *opt-in alternative* for richer/varied voices; `say` remains the deterministic default. Don't wire it into CI.

**Effort:** Small (one file, ~120 lines, mirrors an existing function). No runtime-code risk.

## #909 — cross-episode person identity: "what X said across episodes" → **viewer surface**

**The backend is already built** (payoff of #875/#876). Verified:
- Person identity is **corpus-stable**: `person_id(name) → person:{slug}` is deterministic and shared across episodes ([identity/slugify.py](../../src/podcast_scraper/identity/slugify.py)); the corpus-graph union merges `person:{slug}` into one node across episodes; `#852` identity_map handles spelling variants.
- Cross-episode **queries already exist** in `server/cil_queries.py`: `person_profile()` (insights-by-topic + quotes across all episodes), `position_arc()` (a person's stance on a topic over time), `who_said()`, `topic_person_ids()`.
- **HTTP routes exist**: `GET /persons/{id}/brief`, `/persons/{id}/positions`, `/persons/{id}/topics`, `/topics/{id}/persons` ([server/routes/cil.py](../../src/podcast_scraper/server/routes/cil.py)); plus dashboard `GET /api/corpus/persons/top` (corpus-wide ranking).

**What's missing is the surface** — the viewer doesn't expose the corpus-wide view:
- `PersonLandingView.vue` exists but its `positionRows` are **episode-scoped** (read from the loaded merged artifact), while the corpus-wide data is only reachable via the relational/CIL API.
- No dedicated "click a person → full cross-episode profile page" route.
- SPOKEN_BY graph edges have no dedicated styling.

**Plan (frontend-led; verify backend as step 0):**
0. **Verify** the CIL routes return corpus-wide results against the prod-v2 corpus (don't trust the audit blindly): hit `/persons/{id}/brief` + `/persons/{id}/positions` and confirm multi-episode aggregation. Fix any gap server-side first.
1. **Person profile page**: a routed view (`/person/{person_id}`) that calls `/persons/{id}/brief` (+ `/positions`) and renders: name/aliases, episodes-appeared-in list, topics discussed, and **quotes grouped across episodes** (the headline payoff).
2. **Wire entry points**: dashboard person card + graph Person node → the new page.
3. **Enhance `PersonLandingView`** to pull corpus-wide positions from the API instead of the single-episode artifact (or fold it into the new page).
4. **Graph polish**: SPOKEN_BY edge style in `cyGraphStylesheet.ts`.
5. Tests: viewer unit/e2e for the new view; a server test asserting cross-episode aggregation if step 0 finds a gap.

**Effort:** Medium, mostly Vue/TS in `web/gi-kg-viewer` + a small amount of server verification. Needs the viewer build/test loop (`cd web/gi-kg-viewer`, vitest/playwright) and likely `make ci-ui-full` before push (touches viewer surfaces).
