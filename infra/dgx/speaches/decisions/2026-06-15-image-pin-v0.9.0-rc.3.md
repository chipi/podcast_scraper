# Speaches image pin — `v0.9.0-rc.3-cuda`

**Decision date:** 2026-06-15
**Issue:** #920
**Previous pin:** `:latest-cuda` (floating)
**New pin:** `ghcr.io/speaches-ai/speaches:v0.9.0-rc.3-cuda`

## Context

The Speaches container that backs the DGX `tailnet_dgx_whisper`
transcription path was previously pulled as `:latest-cuda` — a floating
tag. Two `make dgx-deploy` invocations at different times could land
different upstream versions with no audit trail. That's fine for early
iteration, but it makes "DGX is down" debugging much harder than it
needs to be, and it makes prod use rest on a continuously-changing base.

Per #920, this directory adopts the homelab autoresearch vLLM "decisions/"
pattern: every image pin lands a markdown entry here capturing what was
chosen, what was tested, and what the rollback path is.

## What this pin is

`v0.9.0-rc.3-cuda` (published 2025-12-27 by `speaches-ai/speaches`).
Despite the `-rc.3` suffix, upstream marks this release
`prerelease: false` in their GitHub Releases API — they're shipping it
as stable. It's their latest available tag.

## Why not an earlier `v0.8.x`

`v0.8.3-cuda` (2025-09-19) is the most recent truly RC-suffix-free tag.
Chosen `rc.3` over `v0.8.3` because:

1. Upstream itself signals `prerelease: false` on `rc.3`, treating it as
   stable.
2. The original #920 ticket cited `v0.9.0-rc.3-cuda` as the canonical
   pin example.
3. Sticking on `v0.8.3` would mean carrying a 9-months-stale base when
   upstream has already moved on; we'd be pinning to a known-stale
   version, not a known-stable one.

## What didn't change

- `WHISPER__COMPUTE_TYPE=int8` — still pinned per #957. Re-confirmed
  empirically against this `v0.9.0-rc.3-cuda` image in #948 (see
  sibling `2026-06-15-compute-type-int8.md`). All compute types now
  load and produce coherent output on this image (`fp16`/`bf16` no
  longer hard-error — CTranslate2 4.8.0 has Blackwell sm_120 support),
  but `int8` is the fastest by a wide margin (4.89× realtime vs
  fp16's 1.69×). The original "Blackwell kernels missing" hypothesis
  was specific to the pre-#920 floating `:latest-cuda` image.
- `MODEL = "Systran/faster-whisper-large-v3"` — same model, just served
  by a newer Speaches.
- The #968 Thread B temperature-fallback patch applied via the local
  `Dockerfile` — re-applied on top of the new base. The grep-guard in
  the Dockerfile will fail the deploy loudly if the upstream patch
  target moved between versions.

## Roll-forward / rollback procedure

**To bump:**

1. Add a new file under this directory:
   `YYYY-MM-DD-image-pin-vX.Y.Z.md`, copying this template. Include any
   bench numbers from the candidate (cold-boot time, sustained req rate,
   WER on the v2 fixture bed if available).
2. Edit `infra/dgx/converge/deploy.py`:
   `BASE_IMAGE = "ghcr.io/speaches-ai/speaches:vX.Y.Z-cuda"`
3. `make dgx-deploy` from the laptop. The pre-warm step (#920) will
   trigger the new image's first model load; if anything's broken at
   the model-loading layer (e.g. CTranslate2 swapped its compute type
   names), the pre-warm fails visibly. The compose healthcheck (also
   #920) will then mark the container unhealthy until the model is
   actually responsive.

**To roll back:** revert the `deploy.py` edit and re-run `make dgx-deploy`.
The previous tag's image will pull again (assuming GHCR retention has it
— upstream typically keeps tags indefinitely) and the container will
restart on it.

## Vendor watch — when to revisit

- A `v1.0.0` stable from upstream (signal: the `-rc` suffix drops).
- A future Speaches / CTranslate2 release with faster fp16/bf16 kernels
  on Blackwell. Today these kernels work but are slower than int8 on
  this image (#948); a future release where fp16 beats int8 on
  realtime multiple would be a reason to revisit.
- A vulnerability disclosure on the bundled `faster-whisper` /
  CTranslate2 / ONNX Runtime stack.

This isn't a calendar reminder — just a list of signals to look for
next time the operator happens to be reviewing the DGX stack.
