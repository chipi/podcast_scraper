## CRITICAL RULES (MUST FOLLOW - NO EXCEPTIONS)

### Commit Workflow (MANDATORY)

1. **Never:** NEVER commit without showing `git status` first
2. **Never:** NEVER commit without showing `git diff` first
3. **Never:** NEVER commit without explicit user approval
4. ALWAYS show changes → wait for approval → get commit message → THEN commit

### PR Push Workflow (MANDATORY)

1. **Never:** NEVER push without showing `git status` and `git diff` first
2. **Never:** NEVER push without explicit user approval
3. Show changes → ask approval → THEN push
4. Pre-commit hook already ran `make ci-fast` checks, so no need to run `make ci` again

### File Location Rule (MANDATORY)

- ALWAYS create analysis/plan documents in `docs/wip/`
- **Never:** NEVER create `docs/analysis/` or `docs/plan/` folders

### Testing Before Commit (MANDATORY)

1. Run `make ci-fast` before committing when needed (fast validation: formatting, linting, fast tests). For **viewer-heavy** changes under `web/gi-kg-viewer/`, prefer **`make ci-ui-fast`** locally (Playwright; skips Python **`tests/e2e/`**). Pre-commit still runs **`make ci-fast`**. For **docs-only** changes (markdown under `docs/`, no `src/` or app code), prefer **`make fix-md`**, **`make lint-markdown`**, and **`make docs`** instead of insisting on a full **`make ci-fast`** — see **`.cursorrules` rule 5c** (incremental validation) and the **`markdown-pre-commit`** skill.
2. **Skip a duplicate run** if `ci-fast`, `ci-ui-fast`, or `make ci` already passed in-session on the **same** diff with no substantive edits after (say so briefly); see `.cursorrules` rule 5
3. Pre-commit hook will also run `make ci-fast` unless `git commit --no-verify`
4. After pre-commit passes, push directly (no need to run `make ci` again on the same validated diff)

### GitHub Operations (MANDATORY)

1. ALWAYS use MCP GitHub server tools first for all GitHub operations
2. Fallback to GitHub API only if MCP server unavailable or fails
3. **Never:** NEVER use GitHub API directly when MCP server is available
4. Use MCP for: PRs, issues, branches, searches, reviews, comments

## Quick Commands

```bash
make test-ci-fast      # Fast tests (default)
make ci-ui-fast        # Like ci-fast but Playwright instead of Python tests/e2e (viewer work)
make ci                # Full CI validation (before push)
make format            # Auto-format code
make fix-md            # Auto-fix markdown
make lint-markdown     # Check markdown
make docs              # Validate docs build (REQUIRED after docs changes)
```

## Key Patterns

- **Module boundaries**: Respect separation (no HTTP in CLI, no business logic in config)
- **Lazy loading**: Use `__getattr__` pattern for optional dependencies
- **Mock external deps**: Always mock HTTP, ML models, file I/O in unit tests
- **Config-driven**: All runtime options flow through `Config` model
- **Documentation**: Update README for CLI changes, docs/architecture/ARCHITECTURE.md for design changes

## Decision Trees

**New feature?**

- User-facing → Create PRD in `docs/prd/`
- Architecture change → Create RFC in `docs/rfc/`
- Small change (<100 lines) → Skip PRD/RFC

**Where to put test?**

- Single function, fully mocked → `tests/unit/`
- Component interactions → `tests/integration/`
- Complete workflow → `tests/e2e/`

**What tests to run?**

- Any code change → `make test-ci-fast` first
- ML model changes → Also run slow tests
- Before push → `make ci` (mandatory)
